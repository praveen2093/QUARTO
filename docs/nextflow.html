Nextflow — Professional Tutorial for Bioinformatics & Cloud

Industry-standard Nextflow guide for scalable, reproducible pipelines in bioinformatics, pharma, and cloud.
Note

Audience: Students, bioinformatics researchers, clinical programmers, data scientists, pharmaceutical professionals, and software engineers.

1. Introduction

Nextflow is a workflow management system for running computational pipelines. It connects analysis steps, runs them in parallel, and scales from a laptop to the cloud.

Why Nextflow exists

Bioinformatics and data pipelines often involve many steps, large datasets, and many samples. Shell scripts become hard to maintain, restart, and scale. Nextflow was built to solve reproducible, portable, parallel workflow execution.

The problem Nextflow solves

Problem Without Nextflow With Nextflow
Parallelism Manual job arrays Automatic parallel tasks
Failures Restart from scratch Resume from last successful step
Portability Scripts tied to one cluster Profiles for local, AWS, GCP
Reproducibility Ad-hoc environments Docker/Singularity/Conda per step
Complexity Spaghetti bash scripts Structured processes and channels

Where Nextflow is used in industry

  • Academic genomics cores — RNA-seq, WGS, single-cell pipelines
  • Pharma bioinformatics — target discovery, biomarker pipelines
  • CROs & sequencing vendors — standardized nf-core pipelines
  • Public health — pathogen surveillance (e.g., EPI2ME, viral pipelines)
  • Cloud-native labs — AWS Batch, Google Life Sciences, Kubernetes

2. Learning Objectives

By the end of this tutorial, you will be able to

  2. Install Nextflow and run a hello-world workflow locally

3. Why is this important?

Why should someone learn Nextflow?

Nextflow is the de facto standard for portable bioinformatics pipelines. nf-core alone has 100+ community pipelines used worldwide.

Why do companies use it?

  • Scale — process hundreds of samples in parallel
  • Resume — -resume saves time and cloud cost after failures
  • Portability — same pipeline on laptop, HPC, or cloud via profiles
  • Community — nf-core, Seqera Tower ecosystem

Why is it important in bioinformatics?

Genomics workflows are multi-step and data-heavy. Nextflow manages dependencies, parallelization, and containerized tools automatically.

Why is it important in clinical programming?

While SAS/R dominate TLFs, Nextflow supports omics sub-studies, biomarker pipelines, and reproducible secondary analyses linked to trials.

Why is it useful in data science?

Nextflow orchestrates feature engineering, model training, and batch scoring at scale — especially when combined with Docker and cloud executors.

6. Basic Concepts

Process

Definition A single computational step with inputs, outputs, and a script
Analogy One station on a factory assembly line
Industry example FastQC on each FASTQ file in parallel
Best practice One logical task per process; pin container versions

Channel

Definition An asynchronous queue connecting processes with data
Analogy A conveyor belt carrying work items
Industry example One channel item per patient sample
Best practice Use tuple channels for paired-end reads

Workflow

Definition The DAG (graph) linking processes
Analogy The master floor plan of the factory
Industry example QC → align → call variants → annotate
Best practice Keep main.nf readable; modularize with include

Executor

Definition Backend that runs tasks (local, slurm, awsbatch, etc.)
Analogy The HR department assigning workers to stations
Industry example awsbatch for cloud scale
Best practice Use profiles in nextflow.config

Profile

Definition Named configuration bundle (executor, containers, paths)
Analogy A preset mode: “laptop” vs “cloud”
Industry example -profile docker,aws
Best practice Never hardcode credentials in profiles

Container directive

Definition Docker/Singularity image used by a process
Analogy The standardized toolbox for that station
Industry example biocontainers/fastqc:v0.11.9_cv8
Best practice Always specify full image tags

work/ directory

Definition Cached task directories for resume and debugging
Analogy The workshop floor where in-progress jobs live
Industry example Inspected when a variant-calling step fails
Best practice Use shared workDir on cloud; enable -resume

7. Hands-on Examples

Example 1 — Hello World

Problem: Verify Nextflow runs locally.

Solution: Create main.nf:

#!/usr/bin/env nextflow

process sayHello {
  output:
    path 'hello.txt'

  script:
  """
  echo "Hello from Nextflow" > hello.txt
  """
}

workflow {
  sayHello()
}

Run:

nextflow run main.nf
cat work/*/hello.txt

Expected output: Hello from Nextflow

Example 2 — Parallel sample processing

Problem: Process multiple samples in parallel.

Solution:

params.samples = ['S1', 'S2', 'S3']

process greetSample {
  input:
    val sample

  output:
    stdout

  script:
  """
  echo "Processing sample ${sample}"
  """
}

workflow {
  Channel.fromList(params.samples) | greetSample | view
}

Expected result: Three lines, order may vary (parallel execution).

Example 3 — Dockerized FastQC

Problem: Run FastQC in a reproducible container.

Solution:

process runFastqc {
  container 'biocontainers/fastqc:v0.11.9_cv8'

  input:
    path fastq

  output:
    path "*.html"

  script:
  """
  fastqc ${fastq}
  """
}

workflow {
  Channel.fromPath('*.fastq.gz') | runFastqc
}

Expected result: HTML QC reports per FASTQ file.

Example 4 — Python in Nextflow

Problem: Run a Python script inside a pipeline.

Solution:

process runPython {
  container 'python:3.11-slim'

  input:
    path csv

  output:
    path 'out.txt'

  script:
  """
  python -c "import pandas as pd; df=pd.read_csv('${csv}'); print(df.shape)" > out.txt
  """
}

Example 5 — R analysis step

Problem: Execute analysis.R with Dockerized R.

Solution:

process runR {
  container 'rocker/r-ver:4.3'

  input:
    path input_csv

  output:
    path 'output.csv'

  script:
  """
  Rscript analysis.R ${input_csv} output.csv
  """
}

analysis.R:

args <- commandArgs(trailingOnly = TRUE)
dat <- read.csv(args[1])
write.csv(summary(dat), args[2])

11. Best Practices

DO

  • Use -resume during development
  • Pin container tags on every process
  • Store configs in nextflow.config with profiles
  • Test on small datasets first
  • Publish reports: -with-report, -with-trace, -with-timeline
  • Use nf-core pipelines when applicable
  • Keep secrets in environment variables or secret managers

DON’T

  • Hardcode cloud credentials in Git
  • Run production on :latest containers
  • Ignore failed tasks in work/ without logs
  • Mix dev and prod buckets/workDirs
  • Process PHI without access controls on cloud storage

Performance tips

  • Maximize parallelism with channels
  • Use appropriate cloud instance types per process
  • Store references genomes on shared fast storage

Security tips

  • Private buckets, VPC endpoints, IAM least privilege
  • Encrypt data at rest and in transit
  • Audit Tower/Nextflow run metadata

Optimization tips

  • Cache aggressively with -resume
  • Use errorStrategy and maxRetries wisely
  • Profile with -with-trace before scaling

12. Common Mistakes

Mistake Reason Solution
Java not installed Nextflow requires JVM Install Java 11+
Docker not available Container engine missing Install Docker; enable in config
Empty channel Wrong glob or path Check Channel.fromPath patterns
Permission denied on cloud IAM misconfiguration Fix roles for S3/Batch/GCP
Out of memory Instance too small Increase memory in process or instance
Stale resume cache Changed script but same hash confusion Use -resume carefully; delete work dir if needed
Missing input files Relative path issues Use absolute paths or params.input
Slow local runs Too much parallelism on laptop Limit maxForks in config

13. Troubleshooting Guide

Error Cause Solution
Unable to acquire lock Another Nextflow run active Wait or remove .nextflow.lock safely
Process exceeded memory OOM kill Increase memory directive
Cannot find Java JAVA_HOME unset Install JRE; export JAVA_HOME
Docker not found Docker daemon down Start Docker Desktop
Access Denied (S3/GCS) Credentials/permissions Fix IAM; verify bucket
Task hangs Network/storage issue Check .command.log in work dir
Container pull fails Wrong tag or no network Verify image name; login to registry

Diagnostic commands:

nextflow log
nextflow run main.nf -resume -with-trace trace.txt
# Inspect task folder:
cat work/ab/cd1234*/.command.err
cat work/ab/cd1234*/.command.log

14. Interview Questions

Beginner

Q1: What is Nextflow?
A: A workflow manager for parallel, reproducible data and bioinformatics pipelines.

Q2: What is a process?
A: A pipeline step with defined inputs, outputs, and execution script.

Q3: Why use containers with Nextflow?
A: To pin software versions and ensure reproducibility across environments.

Intermediate

Q4: What does -resume do?
A: Reuses cached successful tasks from previous runs, rerunning only failed or changed steps.

Q5: Explain channels.
A: Queues that pass data between processes asynchronously.

Q6: How do profiles work?
A: Named configuration sets selected via -profile for different executors and settings.

Advanced

Q7: Compare Nextflow with Snakemake.
A: Both orchestrate workflows; Nextflow emphasizes dataflow channels and cloud-native executors; Snakemake uses rule graphs — choice depends on team and ecosystem.

Q8: How would you run Nextflow on AWS Batch?
A: Configure process.executor = 'awsbatch', S3 workDir, IAM roles, and container registry access in nextflow.config.

Q9: How do you handle secrets?
A: Environment variables, AWS Secrets Manager, Vault — never commit keys to Git.

20. Mini Project — Two-Step QC Pipeline

Problem statement

Build a Nextflow pipeline that greets each sample ID and simulates a QC pass — foundation for real FastQC integration.

Architecture

Folder structure

nf-qc-demo/
├── main.nf
├── nextflow.config
└── data/
    └── samples.txt

Implementation (main.nf)

params.samples = file('data/samples.txt').readLines()

process greet {
  input:
    val sample
  output:
    tuple val(sample), path("${sample}.txt")

  script:
  """
  echo "QC passed for ${sample}" > ${sample}.txt
  """
}

workflow {
  Channel.fromList(params.samples) | greet | view
}

data/samples.txt:

SAMPLE001
SAMPLE002
SAMPLE003

Commands

nextflow run main.nf -with-report report.html

Expected output

Three .txt files in work/ and a timeline/report HTML.

Cloud execution (reference)

Google Cloud Platform

// nextflow.config excerpt
profiles {
  gcp {
    process.executor = 'google-lifesciences'
    google {
      project = 'YOUR_PROJECT_ID'
      region  = 'us-central1'
      bucket = 'gs://your-nextflow-bucket'
    }
    workDir = 'gs://your-nextflow-bucket/work'
  }
}
nextflow run main.nf -profile gcp -with-report report.html

AWS Batch

profiles {
  aws {
    process.executor = 'awsbatch'
    aws.region = 'us-east-1'
    workDir = 's3://your-nextflow-bucket/work'
  }
}
nextflow run main.nf -profile aws -with-report report.html
Warning

Replace placeholder project IDs, buckets, and IAM roles with your organization’s approved cloud resources.
